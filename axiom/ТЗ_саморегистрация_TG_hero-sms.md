# ТЗ: саморегистрация Telegram-аккаунтов через SMS-сервис hero-sms

Статус: черновик на согласование. Автор проверил API вживую 2026-07-16 (баланс/цены — реальные вызовы). Реализацию НЕ начинать до «ОК» Василия: это тратит деньги и создаёт аккаунты.

## 1. Зачем (цель)

Перестать зависеть от покупных аккаунтов, которые дохнут лотами (история LZT: 10 из 10 слетели — см. память `lzt-arbitrage-luna`). Вместо покупки — **регистрировать номер самим** через SMS-активацию: номер и 2FA под нашим контролем с нулевой секунды, аккаунт свежий и не унаследовал чужих флагов/банов.

## 2. Честно про «затраты те же, риск 0»

Проверено вживую, без прикрас:

- **Затраты.** ~$0.3–1.1 за SMS-активацию Telegram + прокси. Покупной аккаунт — $1–5. Итог: саморег **дешевле или столько же**. Верно.
- **Риск НЕ ноль**, а другой:
  - уходит риск «купили заранее сожжённый аккаунт» (главная боль LZT) — это плюс;
  - **приходит** риск «свежий аккаунт»: Telegram агрессивно банит только что зарегистрированные номера из пулов активационных сервисов, особенно если они сразу идут в холодную рассылку. Свежаку нужен **прогрев** (у нас есть `warmup.py`) и хороший прокси, иначе слетит в первые дни.
  - **Россия сейчас недоступна** (`country=0` → пусто). Номера будут иностранные (KZ/…); это надо учесть в поле `accounts.country` и подобрать гео-совпадающий прокси, иначе связка «иностранный номер + русский IP» сама по себе подозрительна.

Вывод: саморег — правильный ход для контроля и предсказуемости, но это **не «поставил на автомат и забыл»**. Ценность — в управляемости, а не в бессмертии.

## 3. Что НЕ входит

- Не регистрируем сотнями за раз (это спам-паттерн для Telegram и для самого hero-sms). Порциями, с паузами.
- Не подменяем прогрев: этот модуль доводит аккаунт до «залогинен + 2FA + сохранён», дальше — существующий `warmup.py`.
- Не трогаем WhatsApp (отдельная история).

## 4. Подтверждённые факты API (проверено вживую)

- Базовый URL: `https://hero-sms.com/stubs/handler_api.php`
- Авторизация: query-параметр `api_key` (лежит в `.env` → `HERO_SMS_API_KEY`).
- Протокол: SMS-Activate `handler_api` (обратная совместимость — методы стандартные).
- Код сервиса Telegram: `tg`. Страна — числом (`country=`), `0` = Россия (**сейчас 0 номеров для tg**).
- `action=getBalance` → `ACCESS_BALANCE:0.93` (проверено).
- `action=getPrices&service=tg` → JSON `{страна: {tg: {cost, count, physicalCount}}}` (проверено, цены реальные).

Методы, которые предстоит использовать (стандарт SMS-Activate — свериться с их Scalar-докой перед кодом):

| action | смысл | ключевой ответ |
|---|---|---|
| `getBalance` | остаток | `ACCESS_BALANCE:0.93` |
| `getPrices&service=tg&country=N` | цена/наличие | JSON |
| `getNumber&service=tg&country=N` (или `getNumberV2`) | **арендовать номер (платно)** | `ACCESS_NUMBER:{id}:{phone}` |
| `getStatus&id={id}` | опрос кода | `STATUS_WAIT_CODE` / `STATUS_OK:{code}` / `STATUS_CANCEL` |
| `setStatus&id={id}&status=8` | отменить (вернуть деньги, если код не пришёл) | `ACCESS_CANCEL` |
| `setStatus&id={id}&status=6` | подтвердить (активация удалась) | `ACCESS_ACTIVATION` |

⚠️ `getNumber` — **платный и резервирует номер**. В коде вызывается только на реальном шаге регистрации, никогда в тестах/проверках.

## 5. Архитектура

Два модуля + миграция БД, всё переиспользует уже существующее.

### 5.1 `channels/sms_hero.py` — тонкий клиент API
- `balance() -> float`
- `price(country, service='tg') -> dict`
- `get_number(country, service='tg') -> (activation_id, phone)` — платно
- `poll_code(activation_id, timeout=180) -> str | None` — опрос `getStatus` с паузами, вернуть код или None
- `cancel(activation_id)` / `finish(activation_id)`
- Ретраи на сети/5xx — переиспользовать паттерн из `agent/llm._compat_post`.
- Ключ — из `.env` через `config`. Значение в логи/ответы не печатать.

### 5.2 `channels/phone_register.py` — сценарий регистрации
Полный цикл на один аккаунт (Telethon):
1. Взять свободный **мобильный/резидентный прокси** гео-страны из пула (`proxy_pool.py`) — под датацентр-IP Telegram чаще отклоняет регистрацию.
2. `sms_hero.get_number(country, 'tg')` → номер.
3. Telethon `SendCodeRequest(phone)` со **своими `api_id`/`api_hash`** (у нас поля per-account в `accounts`).
4. `sms_hero.poll_code()` → код; `SignUpRequest`/`SignIn` с кодом.
5. Не пришёл код за таймаут → `cancel()` (деньги возвращаются), номер помечаем как неудачу, следующий.
6. Сразу поставить **облачный пароль 2FA** — переиспользовать `channels/twofa.py` (защита от угона, см. память `account-reclaim-2fa`). Пароль сохранить в `accounts.twofa_password`.
7. Заполнить профиль (имя/аватар) — переиспользовать `channels/identity.py`.
8. `sms_hero.finish()` — подтвердить активацию.
9. Сохранить в `accounts`: `tg_session` (StringSession), `phone`, `api_id/api_hash`, `country`, `proxy`, `twofa_password`, `kind='sim'`, `status='warming'`, `bought_at=now`.
10. Отдать аккаунт в `warmup.py` (не в рассылку сразу!).

Антибан-правила (в коде, не «на память»):
- потолок регистраций в сутки — маленький (напр. `--per 3`), с большими паузами;
- один прокси/IP — не подряд много номеров;
- любой аккаунт после регистрации минимум N дней только прогрев, `protected=0`, но в рассылку не берётся, пока `warm_stage` не дорос;
- ошибки Telegram про сессию/бан прогонять через `session_check.note_failure` (уже есть).

### 5.3 Миграция БД
`accounts` в основном готов (`api_id/api_hash/country/proxy/twofa_password/kind/bought_at/session_*` — уже есть). Добавить:
- `reg_source TEXT` — откуда аккаунт: `hero-sms` / `bought` / `own`;
- `reg_activation_id TEXT` — id активации hero-sms (для разбора/возвратов);
- `reg_cost REAL` — сколько стоила активация (для экономики).

## 6. UI (раздел «Аккаунты»)
- Кнопка «➕ Зарегистрировать номер (hero-sms)»: выбор страны (из `getPrices` — показать цену и наличие), сколько штук, каким прокси-пулом. Идёт фоном, как массовый скан: прогресс + стоп.
- Плашка баланса hero-sms (из `getBalance`) с предупреждением, если баланса не хватает на заказанное количество.
- В карточке аккаунта — источник (`reg_source`) и стоимость.

## 7. Критерии приёмки
- `getBalance`/`getPrices` показываются в UI, ключ нигде не светится.
- «Зарегистрировать 1 номер» → в `accounts` появляется живой аккаунт: `session_check` говорит `alive`, стоит 2FA, `status='warming'`, привязан прокси.
- Неудачная активация (код не пришёл) → деньги возвращены (`cancel`), мусорный аккаунт не создан.
- Свежий аккаунт **не попадает в рассылку** автоматически, пока не прогрет.

## 8. Этапы
1. `sms_hero.py` + вывод баланса/цен в UI (read-only, денег не тратит) — можно делать сразу.
2. `phone_register.py`: регистрация ОДНОГО номера под контролем, с 2FA и прогревом.
3. Массовый режим (пачки, потолки, гео-прокси) + UI-кнопка с прогрессом.
4. Экономика в разделе кампаний (стоимость аккаунта → в ROI).

## 9. Открытые вопросы к Василию
- Страна номеров: раз РФ недоступна — какие берём (KZ/иное)? Влияет на прокси и на «легенду» аккаунта.
- Общие `api_id/api_hash` на все аккаунты или свои на каждый? Свои безопаснее, но их надо где-то брать.
- Целевой объём: сколько аккаунтов в неделю? От этого — потолки и размер прокси-пула.
